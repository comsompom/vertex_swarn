Mission — What we achieve

  • Zero central command. No orchestrator, no single point of failure. Machines coordinate directly over Vertex/FoxMQ.
  • Heterogeneous fleet. Sentries (sector patrol) and drones (recon, handoff). Scale at runtime from the dashboard.
  • P2P emergency stop. One message: entire fleet freezes in under 50 ms. No master to approve — pure peer-to-peer safety.
  • AI-driven coordination. Strategies for handoff, sector rebalance, stale recovery. Recommendations published to the mesh.
  • Resilience under failure. Kill nodes; the swarm keeps running. Graceful degradation, no central brain to crash.

How we built it

Transport FoxMQ (Vertex-backed MQTT). Byzantine-fault-tolerant broker; coordination layer is Vertex, not a single server.

Nodes Python agents: sentries (sector, patrol), drones (battery, handoff), spectator (observability). All publish state and subscribe to E-Stop and Unstop.

Dashboard Flask app: live swarm state, mission logo, Logs (operation log), Add nodes, AI Control (run strategies; result goes to the mesh). E-Stop shown as centered red modal when fleet frozen; E-Stop/Unstop from CLI.

Safety E-Stop and Unstop topics. Any node or operator can trigger; every subscriber freezes or resumes. No central controller.

What you're seeing: A perimeter-defence system with no middleman. Coordination, automation, and security are distributed.

Three pillars

Coordinate

Auto-discovery via shared mesh. State sync over topics. Sentries and drones see each other; sectors and roles are shared, not commanded from above.

Automate

Patrol, recon, low-battery handoff. AI Control suggests actions; nodes react. Chaos monkey proves the swarm degrades gracefully when units are lost.

Secure

P2P E-Stop. One trigger, fleet freeze. No approval chain — critical for perimeter defence where latency kills.

Track 1 — Perimeter defence swarm

What we achieved: A multi-drone, multi-sentry swarm over FoxMQ (Vertex). Default: 3 sentries, 3 drones, 1 spectator. Add more from the dashboard. Logs record add node, AI control, E-Stop/Unstop. AI Control runs strategies (handoff, rebalance, stale recovery, OpenAI tactical) and publishes to the mesh. No single point of failure.

What you're seeing now: A live perimeter-defence scenario. Heterogeneous roles, shared state, and P2P safety — all without a central command post.

Live — Swarm over Vertex/FoxMQ

Sentries and drones are running. They connect to the same FoxMQ broker (Vertex-backed). Each node publishes its state; each subscribes to state, E-Stop, and Unstop. The spectator prints the consolidated view. No database, no cloud — just the mesh.

What you're seeing now: The terminal (and/or dashboard) shows the live swarm: node IDs, roles, sectors, battery, last seen. This is the coordinated picture every participant shares. No master node holds the truth.

Live — Dashboard: Add nodes & AI Control

The dashboard is a view into the same mesh. It displays all nodes, E-Stop status (as a centered red modal when frozen), and the latest AI suggestion. Logs (button next to the logo) shows the operation log (add node, AI control, E-Stop/Unstop). Add nodes spawns new drones or sentries. AI Control runs a strategy (e.g. sector rebalance, low-battery handoff); the recommendation is published to the mesh. E-Stop and Unstop are triggered from the CLI. Still no central orchestrator — the dashboard is just another consumer and a spawner.

What you're seeing now: The web UI with the swarm table, mission logo, Logs button, Add nodes, and AI Control. We can grow the fleet and run coordination strategies from the dashboard; E-Stop/Unstop from the terminal; the logic remains P2P.

Resilience — E-Stop, Unstop & chaos

E-Stop: One message on the E-Stop topic. Every node that subscribes freezes. Target: under 50 ms. No central controller — any node or operator can trigger. Unstop: One message on the Unstop topic; the fleet resumes. Trigger from CLI (unstop_trigger.py). The dashboard shows a centered red modal when frozen and logs E-Stop/Unstop in Logs.

Chaos: We kill random nodes from the CLI. The rest keep publishing. In a full deployment they would reallocate sectors and tasks. Here we prove: no master to crash. The swarm degrades; it does not collapse.

What you're seeing now: Either the fleet frozen (E-Stop), resumed (Unstop), or the aftermath of node loss — remaining units still in the mesh. Coordinate, automate, secure. No middleman.
